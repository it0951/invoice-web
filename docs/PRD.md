## 1. 프로젝트 핵심

**목적**: 노션 데이터베이스에 입력된 견적서를 고객이 별도 로그인 없이 공유 링크로 확인하고 PDF로 다운로드할 수 있게 한다.

**사용자**: 프리랜서·소규모 사업자(관리자)와 그들의 고객(클라이언트).

## 2. 사용자 여정

### 관리자 여정

```
1. 관리자 로그인 페이지
   ↓ 이메일 + 비밀번호 입력 후 로그인 버튼 클릭

2. 관리자 대시보드
   ↓ 노션에서 동기화된 견적서 목록 확인
   ↓ 견적서 선택

   [링크 생성/복사 선택] → 공유 링크 클립보드 복사 → 고객에게 전달
   [링크 회수 선택]      → 해당 shareToken 무효화 → 클라이언트 접근 차단
   [목록 계속 관리]      → 관리자 대시보드 유지

3. 공유 링크 생성 완료 → 고객에게 링크 전달
```

### 클라이언트 여정

```
1. 공유 링크 접속 (브라우저 주소창 또는 이메일/메신저 링크)
   ↓ 고유 토큰이 포함된 URL로 자동 진입

2. 견적서 상세 페이지 (토큰 유효성 자동 검증)
   ↓ [토큰 유효 + 미만료 + 미회수] → 견적서 내용 표시
   [토큰 무효/만료/회수됨]         → 오류 유형별 안내 페이지로 리디렉션

3. 견적서 확인
   ↓ 항목별 금액, 합계, 발행일 등 확인

4. PDF 다운로드 버튼 클릭
   ↓ PDF 파일 브라우저 다운로드

5. 완료 → 탭 닫기 또는 계속 열람
```

## 3. 기능 명세

### MVP 핵심 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|--------------|------------|
| **F001** | 노션 DB 견적서 조회 | Notion API로 DB의 견적서 데이터를 읽어 웹에 표시 | 서비스의 유일한 데이터 소스 | 견적서 상세 페이지, 관리자 대시보드 |
| **F002** | 공유 링크 생성 | 견적서별 고유 토큰이 포함된 URL 생성 및 클립보드 복사 | 클라이언트 접근의 유일한 진입점 | 관리자 대시보드 |
| **F003** | 토큰 기반 견적서 열람 | URL의 고유 토큰을 검증(유효성·만료·회수 여부)하고 해당 견적서를 표시 | 로그인 없이 고객이 견적서를 확인하는 핵심 메커니즘 | 견적서 상세 페이지 |
| **F004** | PDF 다운로드 | 견적서를 PDF 파일로 변환하여 다운로드 | 서비스의 최종 가치 전달 | 견적서 상세 페이지 |
| **F005** | 견적서 목록 관리 | 관리자가 노션에서 동기화된 견적서 전체 목록을 확인 | 관리자가 어떤 견적서가 있는지 파악하고 링크를 생성하는 기반 | 관리자 대시보드 |
| **F006** | 공유 링크 회수 | 생성된 shareToken을 무효화하여 클라이언트 접근 차단 | 유출된 링크 또는 만료 처리 | 관리자 대시보드 |

## 6. 데이터 모델

### Invoice (견적서) — Notion DB 스키마

노션 데이터베이스의 각 행(Row)이 하나의 견적서를 나타냅니다.

| 필드 | 설명 | 타입 |
|------|------|------|
| id | 노션 페이지 고유 ID | UUID (Notion 자동생성) |
| title | 견적서 제목 (= 견적 번호) | Title (Notion 기본 필드) |
| clientName | 고객(클라이언트) 이름 | Text |
| clientEmail | 고객 이메일 | Email |
| issueDate | 견적 발행일 | Date |
| status | 견적서 상태 | Select (초안 / 발송 / 확정) |
| totalAmount | 서버 검증용 합계 금액 (quantity × unitPrice 재계산값과 대조) | Number |
| shareToken | 공유 링크용 고유 토큰 (`crypto.randomBytes(32).toString('base64url')`) | Text |
| tokenExpiresAt | 공유 토큰 만료 일시 | Date |
| tokenRevokedAt | 공유 토큰 회수 일시 (설정 시 접근 차단) | Date |
| viewCount | 클라이언트 열람 횟수 (로깅용) | Number |
| lastViewedAt | 마지막 열람 일시 | Date |

> **shareToken 보안 설계**
> - `crypto.randomBytes(32).toString('base64url')` 사용 (256비트 엔트로피)
> - 기본 만료: 발행일로부터 30일
> - 관리자 "링크 회수" 클릭 시 `tokenRevokedAt` 기록 → 이후 접근 차단
> - Notion DB full-scan(filter) 대신 `token → pageId` 매핑을 서버 캐시에 유지

### InvoiceItem (견적 항목) — 별도 연결 Notion DB

> **결정**: "서브아이템" 방식 대신 **별도 InvoiceItem DB + Relation** 방식으로 확정.  
> 이유: 스키마가 명확하고 항목별 수정·정렬·집계가 쉬우며, 쿼리 경로가 단순함.

각 견적서에 포함된 개별 항목입니다.

| 필드 | 설명 | 타입 |
|------|------|------|
| id | 항목 고유 식별자 | UUID (Notion 자동생성) |
| invoiceId | 소속 견적서 ID | Relation → Invoice DB |
| description | 항목명 및 설명 | Title |
| quantity | 수량 | Number |
| unitPrice | 단가 | Number |
| subtotal | 소계 (서버에서 quantity × unitPrice 계산, Notion formula 필드 병행 가능) | Number |

### 서버 캐시 (인메모리, 영속 저장소 없음)

공유 토큰과 Notion 페이지 ID 간의 매핑을 캐싱하여 Notion API 호출을 최소화합니다.

| 필드 | 설명 |
|------|------|
| token | shareToken 값 |
| pageId | 연결된 Notion 견적서 페이지 ID |
| cachedAt | 캐시 저장 일시 (TTL 60초) |

## 7. 기술 스택

### 프론트엔드 프레임워크

- **Next.js 16.2.3** (App Router, Turbopack) — React 풀스택 프레임워크  
  ⚠️ `AGENTS.md` 경고: 기존 Next.js 지식과 다를 수 있음 → 구현 전 `node_modules/next/dist/docs/` 필수 확인
- **TypeScript 5.x** (strict mode, `any` 타입 금지) — 타입 안전성 보장
- **React 19.2.4** — UI 라이브러리

### 스타일링 및 UI

- **TailwindCSS v4** (CSS-first 설정, `@import "tailwindcss"`) — 유틸리티 CSS 프레임워크
- **ShadcnUI** (radix-ui 기반) — 고품질 접근성 보장 컴포넌트
- **Lucide React** — 아이콘
- **`@media print` CSS** — 인쇄용 스타일 (PDF 장애 시 대체 수단)

### 폼 및 검증

- **React Hook Form 7.x** — 폼 상태 관리
- **Zod v4** — 스키마 검증 (로그인 폼, API 응답 파싱, Notion 응답 런타임 검증)

### 서버 상태 관리

- **TanStack React Query v5** — 노션 API 데이터 패칭 및 캐싱 (클라이언트 컴포넌트용)

### 인증

- **Auth.js (NextAuth v5)** — Credentials Provider (이메일/비밀번호 + bcrypt)  
  이유: 자체 세션/CSRF/쿠키 구현 리스크 제거, Next.js 16 App Router 공식 지원

### 데이터 소스

- **Notion API (`@notionhq/client`)** — 견적서 데이터 소스 (백엔드 역할)
- **Next.js Route Handlers** (`app/api/`) — Notion API 키를 서버에서만 사용하는 BFF 레이어
- **Data Access Layer (DAL)** — `import 'server-only'` + `cache()` + DTO 반환으로 Notion raw response 클라이언트 노출 차단

### PDF 생성

- **@react-pdf/renderer** (1순위) — React 컴포넌트로 PDF 구성, `renderToBuffer` API 사용  
  ⚠️ React 19.2.4 호환성 미검증 → **착수 전 스파이크 필수**  
  ⚠️ 한글 폰트 등록 필수: `public/fonts/NotoSansKR-*.ttf` + `Font.register()` 설정
- **`window.print()` + `@media print`** (대체 수단) — @react-pdf 장애 시 즉시 전환 가능

### 피드백

- **Sonner v2** — 토스트 알림 (링크 복사 완료, 링크 회수 완료 등)

### 배포

- **Vercel** — Next.js 최적화 배포 플랫폼  
  환경변수: `NOTION_API_KEY`, `NOTION_INVOICE_DB_ID`, `NOTION_ITEM_DB_ID`, `AUTH_SECRET`  
  Serverless Function 메모리: **512MB 이상** 권장 (한글 TTF 폰트 로드)

## 8. 기술적 위험 및 대응 전략

### Risk 1: Notion API Rate Limit (3 req/s)

| 항목 | 내용 |
|------|------|
| **위험** | 관리자 대시보드 목록 N건 + InvoiceItem 조회 시 N+1 요청 발생 → 429 오류 |
| **대응** | `unstable_cache` / React `cache()` + `revalidate: 60` 적용 |
| **운영** | 관리자 대시보드에 "수동 동기화" 버튼 (강제 캐시 무효화) |
| **캐싱** | token → pageId 매핑 인메모리 캐시 (TTL 60초) |

### Risk 2: @react-pdf/renderer + React 19 호환성

| 항목 | 내용 |
|------|------|
| **위험** | @react-pdf는 자체 reconciler 사용 → React 19와 충돌 가능성 미검증 |
| **대응** | **착수 첫 날** `renderToBuffer` 단순 출력 스파이크 테스트 |
| **fallback** | 실패 시 `window.print()` + `@media print` CSS로 전환 |
| **환경** | Route Handler에 `export const runtime = 'nodejs'` 명시 (Edge Runtime 사용 불가) |

### Risk 3: 한글 폰트 미등록

| 항목 | 내용 |
|------|------|
| **위험** | @react-pdf 기본 폰트(Helvetica)는 한글 미지원 → 빈 박스 출력 |
| **대응** | `public/fonts/NotoSansKR-Regular.ttf` 등 배치 + `Font.register()` 설정 |
| **용량** | TTF 파일 약 4MB → Vercel cold start 영향, 폰트 서브셋 적용 검토 |

### Risk 4: Notion 단일 의존성

| 항목 | 내용 |
|------|------|
| **위험** | Notion API 다운타임 시 서비스 전체 중단 |
| **대응** | 오류 안내 페이지에 "다시 시도" 버튼 + 지수 백오프 재시도 (최대 3회) |
| **모니터링** | Notion API 상태 페이지 확인 권장 |
